Company Registration API

Demo guide · Java 25 · Spring Boot 4 · HTTPS :8443

1. Purpose & clients

A REST API for asynchronous company registration: clients submit a registration, receive a requestId, poll until the job finishes, and optionally rename an active company later. Duplicate checks (DB + vector-search placeholder) run in a background worker, not on the HTTP thread.

Target clients are B2B integrators — partner systems, internal apps, or portals — each with an API key in the clients table:

  • dev-client, partner-client, internal-client (see seed data)
  • Authenticate with header X-API-Key; the server resolves client_id for idempotency and request scoping

Not a browser UI — callers are backend services or automation using JSON over HTTPS.

2a. Registration flow

Client → POST /api/companies → 202 + requestId → Background worker → GET …/requests/{id} poll
  1. Send clientRequestId (UUID per form submit), registrationNumber, name.
  2. API saves a PENDING row → 202 Accepted (or 200 if same clientRequestId replay).
  3. Worker on registrationTaskExecutor: PENDING → PROCESSING, then parallel duplicate lookup (DB + vector), creates/links company → COMPLETED or FAILED.
  4. Client polls GET /api/companies/requests/{requestId} until terminal status.

2b. Update name flow

Client → PUT /api/companies/{registrationNumber} → JSON {"name":"…"} → 200 + CompanyResponse
  • Synchronous — no requestId; updates an ACTIVE company only.
  • INACTIVE → 409; reactivate via registration POST.
  • Append-only row in company_name_history for each rename.

3. Registration request status

PENDING → PROCESSING → COMPLETED
PENDING → PROCESSING → FAILED
StatusMeaning
PENDINGSaved on submit; waiting for worker
PROCESSINGWorker running duplicate lookup / register
COMPLETEDCompany registered or linked (terminal)
FAILEDConflict or error; see errorMessage (terminal)

Transitions are logged in registration_request_status_history (not returned on the public status API).

3b. Parallel duplicate lookup (worker)

Duplicate checks run inside RegistrationRequestWorker via RegistrationLookupOrchestrator — not on the HTTP thread that returned 202. The worker calls resolve().block() and waits for a single LookupOutcome.

Worker thread → vectorSearch.subscribe() ∥ DB lookup Mono → block() → outcome
ComponentRuns onRole
registrationTaskExecutor registration-* threads Background worker per request; calls orchestrator.resolve()
CompanyDbLookupService Reactor boundedElastic Authoritative ACTIVE reg# / name checks (JPA)
VectorSearchService Reactor boundedElastic Similarity search placeholder (app.vector-search.delay-ms)
  1. Build lazy vectorSearch mono (searchByName → map → cache()) — no I/O yet.
  2. vectorSearch.subscribe() starts vector work immediately (parallel, non-blocking for the worker).
  3. DB lookup() runs on another boundedElastic thread when block() subscribes.
  4. DB link or reject → cancel vector subscription → return outcome (DB wins).
  5. DB no match → wait for cached vector result → today always register new company.

Vector errors on the early subscription are logged at WARN only; they do not fail the job when DB already decided. Real vector HTTP should use reactive WebClient so cancel on dispose is reliable.

AOP (ServiceLoggingAspect)

  • Location: src/main/java/.../aop/ServiceLoggingAspect.java
  • Wraps every @Service call (submit, worker process, CompanyService.register, lookup, etc.)
  • Logs: CompanyService.updateName(..) completed in 12 ms
  • Metrics: timer service.method with tags class, method, exception
  • register appears only after the async worker runs (poll until COMPLETED)

Prometheus scrape

Single URL (filter output — do not read the whole page):

https://localhost:8443/actuator/prometheus

  • Service latency: search service_method_seconds + class + method
  • quantile="0.5" / 0.95 / 0.99 = p50 / p95 / p99 (seconds)
  • _count and _sum on the same tags → call count and average
  • HTTP routes: http_server_requests_seconds (histogram buckets)

/actuator/metrics is for quick debugging (no percentiles on timers in Spring Boot 4). Use /actuator/prometheus or Grafana for service performance. SLF4J logs from interceptors + AOP help trace individual requests.
